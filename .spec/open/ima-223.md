# IMA-223 — Deconvolution operator

_slug: op-decon_

> **Status: re-specced by /plan-eng-review on 2026-07-20.** The original spec
> ("plane-op operator (consumes {}) wrapping maragall/deconvolution") is not
> implementable as written — see Decision D2. This document replaces it.

## Why

Deconvolution un-blurs a widefield z-stack by inverting the microscope's point
spread function (PSF). For SquidMIP the payoff is not a deconvolved stack as a
deliverable — it is a **sharper MIP**: run decon over the z-stack as a
pre-filter, then max-project, and out-of-focus haze stops leaking into the
projected plate.

Upstream is `maragall/deconvolution`, whose installed package name is
**`petakit`** (PetaKit5D algorithms: Richardson-Lucy, and OTF-Masked Wiener as
the default). Not blind — it needs a supplied PSF.

## What changed from the original spec, and why

The original spec had three load-bearing errors. All three are verified against
source, not inferred.

**1. Deconvolution cannot be a plane-op.** `petakit.core.deconvolve` hard-raises
on anything that is not a 3D stack:

```python
# deconvolution/src/petakit/core.py:37
if image.ndim != 3:
    raise ValueError(f"Image must be 3D (Z,Y,X), got shape {image.shape}")
if psf.ndim != 3:
    raise ValueError(f"PSF must be 3D (Z,Y,X), got shape {psf.shape}")
```

`consumes {}` means the engine hands the operator one 2D plane at a time, so the
operator as specced raises on its first call, every call. This is not a degraded
mode — the whole value of the OMW/RL method is the *axial* PSF, so plane-by-plane
deconvolution is a different and largely pointless operation.

**2. IMA-210's three-slot taxonomy has no category for it.** IMA-210 proposes
`consumes ∈ {∅, {z}, {fov}}` driving "one group-by-then-reduce engine". Decon
consumes the z-axis as *input* but returns a stack of the **same shape** — a
z-preserving transform, not a reducer. Neither `{}` (2D in) nor `{z}` (one plane
out) can express it.

**3. The engine has exactly one operator slot and a Z=1 write path.**

```python
# squidmip/projection.py:157-162
out = np.empty((n_t, len(channels), 1, y, x), dtype=meta["dtype"])
for t in range(n_t):
    for c_i, channel in enumerate(channels):
        planes = (reader.read(region, fov, channel, z, t) for z in z_levels)
        out[t, c_i, 0] = reduce(planes)  # streamed z; bounded memory
```

`project_well` takes one `reduce=` callable and writes into a hardcoded `Z=1`
buffer. There is no way to say "decon, then MIP".

## Decisions

| # | Decision | Rationale |
|---|---|---|
| **D2** | IMA-210 grows a **transform** category: separate `consumes` from `reduces`. Decon declares `consumes={z}, reduces=∅`. | Fixes the taxonomy once, before siblings 222/224/225 harden the contract against a model that cannot express transforms. |
| **D3** | A run is a **pipeline**: zero or more shape-preserving filters, then exactly one reducer. `project_well` applies filters to the z-stack, then hands the result to `reduce`. | Serves 223 (decon), 224 (bgsub) and 225 (flatfield) with one engine change instead of three special cases. Keeps the Z=1 write path and the zarr writer untouched. |
| **D4** | Plumb `na` into reader metadata; build the PSF from **SquidMIP** metadata and pass **arrays only** to petakit. Never call `petakit.open_acquisition`. | petakit's reader recomputes `dxy = sensor_px / mag`, which SquidMIP explicitly rejects (`reader.py:202`) and whose fixture values differ by 1.7x. A mis-scaled PSF fails silently. |
| **D5** | petakit behind an optional `decon` extra, **imported lazily inside the operator**, CPU path as the tested default. | petakit hard-pins `cupy-cuda12x` (no macOS/CPU wheels); the ship path is a Windows venv. Prior learning `tilefusion-init-heavy-import`: never drag a heavy stack at module import. |

## Blocking dependency — the contract IMA-223 needs from IMA-210

IMA-210 is **Todo, not started** (branch `juliomaragall/ima-210-operator-consumes-axis`
sits at `1504c05`, byte-identical to `main`). IMA-223 cannot start until IMA-210
lands, and IMA-210 must land with these two properties, which its current spec
does not have:

1. **`reduces` is separate from `consumes`.** An operator declares both. A
   filter is `consumes={z}, reduces=∅` (stack in, same-shape stack out). A
   reducer is `consumes={z}, reduces={z}` (stack in, plane out).
2. **`project_well` accepts an ordered `filters=[]` before its single `reduce=`.**

Both belong in IMA-210, not here — that is the whole point of doing it there
before three more operators build against the narrower contract.

The pipeline is deliberately arranged so **everything downstream of `reduce` is
untouched**: the output buffer, `_output.py`, the zarr writer and ndviewer_light
all keep their existing Z=1 contract.

## The memory tradeoff this introduces

Today `project_well` streams z (`planes` is a generator; `project` accumulates
with `np.maximum`), so per-worker memory is one plane. **A filter stage breaks
that** — decon needs the whole stack at once, by construction.

Per FOV per channel, for a 3000×3000×20 uint16 stack:

| Buffer | Size |
|---|---|
| stack as float32 | ~720 MB |
| FFT working set (complex64, several live) | ~1.4 GB each |
| realistic peak | **multi-GB** |

`_default_workers()` (`_engine.py:80`) returns every usable core. Running decon
at that width will OOM the machine. This is handled in T6 below, and it is the
single most likely way this feature fails in production.

Note `add_projector`'s own docstring already anticipates this: *"a projector that
materialises the whole z-stack (e.g. EDF) is allowed but owns its own,
documented, memory profile."*

## Distribution

petakit's `pyproject.toml` lists `cupy-cuda12x` as a **hard** dependency, so a
plain `pip install petakit` fails on macOS and on any CPU-only Windows box —
which is every machine this currently ships to. petakit's own README works around
this with a two-step install, and SquidMIP must document the same:

```bash
pip install --no-deps "petakit @ git+https://github.com/maragall/deconvolution@main"
pip install "numpy>=2.0" tifffile scipy psfmodels psutil
```

`allow-direct-references = true` is already set in `pyproject.toml` for the
`ndviewer_light` git URL, so the extra follows an existing convention.

The clean fix is upstream — move `cupy-cuda12x` to a `[gpu]` extra in
`maragall/deconvolution`. Captured as a TODO; not blocking.

## Acceptance / oracle

The original oracle ("Decon runs as a plane-op on the selected subset") would
have passed while producing scientifically wrong output. Replaced:

1. Running **decon → MIP** on a subset produces a plate whose images are
   measurably sharper than plain MIP (higher Tenengrad gradient energy — the
   metric `project_reference` already uses, `projection.py:83`) on the same data.
2. Output shape, dtype and on-disk layout are **identical** to a plain MIP run;
   the plate still opens in ndviewer_light.
3. Plain MIP with no filters is **byte-identical** to today's output (regression).
4. `import squidmip` does **not** import petakit, cupy, or scipy.fft.
5. With petakit absent, the decon card is disabled with a clear install message —
   never a traceback.
6. With `NA` absent from `acquisition.yaml`, the run fails **loud** with a message
   naming the missing field — never a default-NA silent guess.
7. The PSF is generated once per `(channel, nz)`, not once per FOV.

## NOT in scope

- **Deconvolved z-stacks as a saved artifact.** ~20x plate storage, and
  `_output.py` assumes Z=1. Decon is a pre-filter to MIP (D3).
- **GPU/CUDA support as a tested path.** CPU is what ships and what CI runs;
  petakit falls back silently, so GPU works if present but is not validated here.
- **Chaining decon with bgsub/flatfield.** The pipeline makes it *possible*;
  proving the combinations belongs to 224/225.
- **IMA-210 itself.** D2/D3 define the contract 223 needs; building it is 210's job.
- **A params UI schema.** `Operation` has no params mechanism; decon uses the
  existing `_build_run_tab` subset path with method/iterations defaulted.
- **Upstream cupy fix in `maragall/deconvolution`.** Different repo; TODO'd.
