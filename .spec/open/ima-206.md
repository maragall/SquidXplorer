# IMA-206 — Per-channel plate preview

_slug: per-channel-preview_

## Why

P1 (verbatim): one PNG per channel; toggle and contrast adjustment swaps the plate mosaic.

The plate overview composites every channel into one RGB image. That is the right default and
the wrong only option: a 638 signal sitting under a bright 405 is invisible, and there is no way
to ask "show me just this channel, windowed the way I want". This ticket makes the channel axis
and the contrast window user-controlled, on the live plate and in the exported artifact.

## Scope

Render the plate per channel, let the user toggle which channels show and re-window each one
without recomputing anything, and export the same thing as one PNG per channel.

Two deliverables (D2):

1. **Live (Qt)** — `PlateOverview` owns a per-channel plate canvas; channel checkboxes and
   per-channel contrast sliders recomposite from retained pixels. No re-read, no re-projection.
2. **Export (PNG + HTML)** — `build_montage(per_channel=True)` writes `plate_montage_<channel>.png`
   alongside the composite, and the existing self-contained HTML viewer gains channel toggles.

Live compositing is the superset: the same `composite()` serves both.

## Decisions

Each line is choice — why — what was rejected. D1-D11 are the review's interactive decisions;
OV-* are refinements folded in from the outside voice (a Claude subagent; Codex not installed).

- **D2 — Build both the live toggle and the per-channel PNGs.** P1 literally says "one PNG per
  channel", and `_montage.py:340` already holds a per-channel `(C, H, W)` canvas before it
  composites, so the export half is ~50 lines. Rejected: viewer-only (drops what P1 asked for);
  montage-only (drops the interaction everyone demos).
- **D3 — The per-channel store lives in `PlateOverview`, not in a worker.** Only
  `_OperatorWorker` retains per-channel pixels today (`_viewer.py:807, :830`); `_PreviewWorker`
  (`:935-941`) and `_ComputedPlateWorker` (`:984-989`) composite straight to uint8 RGB and throw
  the channel axis away. One store in the widget means the toggle works on open and on reopen,
  not only after a MIP run. Rejected: per-worker stores (triplicates state); operator-only
  (feature dead on the two most common entry paths).
- **D4 — Per-channel auto/manual contrast latch.** `_RunningContrast.window()` (`:504-512`)
  recomputes from a growing histogram and every producer re-reads it per well, so a manual
  slider would be stomped by the next well that lands. First drag on a channel latches it
  manual; untouched channels keep auto-scaling; a per-channel "auto" button unlatches.
  Rejected: freeze-everything-on-first-edit (stops improving channels the user never touched);
  contrast-only-after-run (no control during the long run, which is when you want it).
- **D5 — The Qt half waits for IMA-187; the export half does not** (refined by OV2, below).
  IMA-187 turns a well cell into a multi-FOV sub-grid, which changes what `add_tile` (`:582`)
  carries. Designing the widget store against one-tile-per-well means rewriting it.
  Rejected: build-now-and-migrate (guarantees a second pass through a file three branches edit).
- **D6 — The reopen path uses the global window too.** `_ComputedPlateWorker` takes percentiles
  per well (`:987`) while the other two producers share one global per-channel histogram
  (`:828`, `:938`). Same plate, different picture, and wells that cannot be compared — which
  `_montage.py:29-31` calls out as the whole point of a plate overview. Fixed here because the
  contrast slider has no global window to sit on otherwise. Rejected: separate ticket (ships the
  slider before the path it depends on is consistent); leave as-is.
- **D7 — One `composite()` used by all five sites.** The window-multiply-sum loop is written at
  `_viewer.py:825-829`, `:889-891`, `:936-939`, `:985-988` and `_montage.py:373-381`. What is
  duplicated is exactly what this ticket makes variable: which channels are on and what window
  each uses. Rejected: viewer-only collapse (leaves the export free to drift from the screen).
- **D8 — Rewrite the memory paragraph in the `_viewer.py` module docstring.** `:33` says
  "88x88xC **float32** … ~190 MB for a 1536wp"; `:822` allocates native dtype ("half the RAM"),
  so it is ~95 MB. This ticket makes that store load-bearing and adds to the same budget.
- **D9 — Exported HTML gets channel toggles, no contrast slider.** A PNG is 8-bit,
  already-windowed pixels; re-windowing in the browser cannot bring back crushed shadows, and a
  control that silently degrades the image is worse than no control in a scientific tool. The
  sidecar already records the per-channel window it was rendered at (`_montage.py:395-397`) —
  show it. Rejected: 16-bit + canvas windowing (deferred, see TODOS); 8-bit "preview only".
- **D10 — Cover all 19 identified gaps**, including the assertion that a toggle triggers no
  recompute. A display feature with no proof it avoids recompute has not been verified.
- **D11 — The store is one `(C, H, W)` plate-sized array**, the layout `_montage.py:340`
  already proved at plate scale — not a dict of per-well tiles. Same bytes (~95 MB at
  1536wp × 4ch uint16); a re-window becomes C vectorized ops instead of 6144 small ones.
  Rejected: per-well dict (a slider tick costs a 143 MB grid rebuild + a 12 MP smooth rescale).

Folded in from the outside voice, resolved under delegated authority (the user handed the plan
back with "you don't need my input anymore"):

- **OV1 — The acceptance oracle as written was wrong.** "Colors match the fallback map"
  contradicts `_channels.py:134-158`, whose documented precedence is YAML `display_color` →
  `CHANNEL_COLORS_MAP` → raise. Testing against the fallback map would enforce a regression for
  any acquisition whose YAML specifies its own colors. The oracle now tests the **resolved**
  `meta["channels"][i]["display_color"]`, which is what all five compositing sites already
  consume. `_channels.py` needs no change and is dropped from Files; `_montage.py` is added.
- **OV2 — Split the two halves into independent lanes.** The export half touches only
  `_montage.py` and has zero dependency on IMA-187. Serializing it behind multi-FOV delays the
  shippable half for nothing. Lane A (export) starts now; Lane B (Qt) waits on 187.
- **OV3 — D5 and D11 only look contradictory.** The store is the flat `(C, H, W)` plate canvas;
  the FOV axis is *placement within a well's cell region*, not a key. `_montage.py:356-360`
  already corner-places a tile by its actual shape, which is the same operation a FOV sub-grid
  needs. Post-187 the canvas grows with the sub-grid, not with a new dimension.
- **OV4 — Recomposite ownership is assigned.** Full-resolution recomposite never runs on a
  slider tick. See the recomposite pipeline diagram below.
- **OV5 — Composite at display resolution.** The widget already caches `_scaled` per zoom level
  (`:697-700`); windowing the downsampled view costs a fraction of the full grid.
- **OV6 — Layer × channel memory multiplies.** Each layer that supports toggling needs its own
  `(C, H, W)` store. Two layers (raw + one operator) at 1536wp × 4ch uint16 is ~190 MB. D8's
  rewritten paragraph must state this, and the store must be allocated per layer, lazily.
- **OV7 — The other two producers need wiring, not just a feed.** `_PreviewWorker` has only a
  `tileReady` signal and `_ComputedPlateWorker` has no `finalReady` at all (`:906`, `:953-957`).
  Feeding the store is necessary but not sufficient: both need an end-of-stream recomposite, or
  early wells stay windowed by an immature histogram forever.
- **OV8 — Toggle is a multi-select mask, not an exclusive swap.** P1's "swaps the plate mosaic"
  is the degenerate one-channel-on case of a mask. Checkboxes, all on by default.
- **OV9 — A single visible channel stays tinted with its LUT color**, not grayscale. This
  matches the acceptance oracle and the existing machinery. Many microscopists prefer grayscale
  for single-channel inspection — captured as a TODO rather than guessed at here.
- **OV10 — "No recompute" is defined**, so the test is not vibes: no reader I/O and no
  projection. Recompositing from the retained store is exactly what is allowed.
- **OV11 — Keep the QImage backing array alive.** `_set_final` (`:1854-1859`) holds
  `self._final_arr` because `QImage` wraps the numpy buffer without copying. Every new
  recomposite path must do the same or it is a use-after-free, not a bug.

## Architecture

Where the channel axis survives today, and where it dies (the reason for D3):

```
  PRODUCER                         WHAT IT EMITS              CHANNEL AXIS
  ─────────────────────────────    ──────────────────────     ────────────
  _OperatorWorker._on_well :816    uint8 RGB tile   ──┐        retained in _raw :830  ✓
  _PreviewWorker.run       :920    uint8 RGB tile   ──┼──►     DISCARDED at :941      ✗
  _ComputedPlateWorker.run :977    uint8 RGB tile   ──┘        DISCARDED at :989      ✗
                                                     │
                                       PlateOverview.add_tile :582
                                       (accepts only finished RGB)
```

After this ticket — one store, one compositor, three thin producers:

```
  _OperatorWorker  ─┐
  _PreviewWorker   ─┼──►  add_tile(ri, ci, fov, tile:(C,cell,cell) native dtype)
  _ComputedPlateWorker ─┘                 │
                                          ▼
                        PlateOverview._store[layer] : (C, H, W) native   ◄── D11, OV3
                        PlateOverview._contrast     : per-channel window + latch  ◄── D4
                                          │
                                          ▼
                        composite(store, colors, windows, mask) -> RGB   ◄── D7
                                          │
                            ┌─────────────┴─────────────┐
                            ▼                           ▼
                   Qt canvas (live)          build_montage(per_channel=True)
                                                  plate_montage.png
                                                  plate_montage_<ch>.png  ◄── D2
                                                  plate_montage.html (toggles)  ◄── D9
```

Recomposite pipeline — this is what keeps the slider smooth (OV4, OV5):

```
  EVENT                    RESOLUTION        THREAD      WHEN
  ───────────────────      ────────────      ──────      ──────────────────────────
  tile arrives             cell only         GUI slot    write into store, blit 1 cell
  slider drag (live)       DISPLAY (_scaled) GUI         coalesced: max 1 per frame
  slider release           DISPLAY           GUI         + schedule full-res
  channel toggle           DISPLAY           GUI         immediate (mask is cheap)
  run end / stream end     FULL (C,H,W)      worker      once, then set_final()  ◄── OV7
  zoom change              DISPLAY           GUI         existing _scaled cache path
```

The invariant: a user gesture never touches the full `(C, 2816, 4224)` canvas. The full-res
composite happens once per stream, off the GUI thread, exactly as `_final_montage()` does today.

## Acceptance / oracle

- Toggling a channel off removes exactly that channel's contribution from the plate; toggling
  all but one leaves the single-channel mosaic, tinted with its LUT color (OV8, OV9).
- Dragging a channel's contrast re-windows the plate in place, with **no reader I/O and no
  projection** — recompositing from the store is what is expected (OV10).
- A latched channel keeps its manual window while wells continue to stream; an unlatched channel
  keeps auto-scaling; reset-to-auto restores the running window (D4).
- Colors match the **resolved** `display_color` — YAML first, `CHANNEL_COLORS_MAP` as fallback,
  raise if neither resolves (OV1, `_channels.py:134-158`).
- `build_montage(per_channel=True)` writes one PNG per channel plus the composite; each channel
  PNG carries only that channel, in its color, at the window recorded in the sidecar.
- A reopened computed plate shows the same relative brightness across wells as the run that
  produced it (D6).

## Test plan

Framework: pytest (`pyproject.toml:50`), Qt widgets headless under `QT_QPA_PLATFORM=offscreen` —
the harness `tests/test_viewer.py` already uses.

**CRITICAL (regression, D6):** a reopened plate windows wells globally. Two wells of clearly
different brightness must keep their relative brightness on the reopen path, mirroring
`tests/test_montage.py:144`. This is a regression test: `_ComputedPlateWorker:987` currently
normalizes each well independently, so wells that differ look identical.

Pure functions — `tests/test_montage.py`:
- `composite()` with a mask excluding a channel: that channel contributes nothing.
- `composite()` with an empty mask: all black, no divide-by-zero, no NaN.
- `composite()` applies a distinct window per channel.
- `build_montage(per_channel=True)`: one PNG per channel + the composite; filenames in sidecar.
- each per-channel PNG is dominated by its own `display_color`, mirroring `:123`.
- sidecar records the per-channel window used for each exported PNG.
- resolved-color precedence: a YAML `display_color` beats `CHANNEL_COLORS_MAP` (OV1 guard).

Widget — `tests/test_viewer.py`:
- `add_tile` stores the per-channel tile and composites through `composite()`.
- toggle: canvas changes, and **the reader is never called and no projection runs** (OV10) —
  assert via a spy on the reader, not by timing.
- re-window: canvas changes, store bytes unchanged.
- latch: after a manual drag on channel 0, a newly arriving well moves channel 1's window and
  leaves channel 0's alone (D4).
- reset-to-auto restores the running window.
- all channels off: black plate, no crash.
- single-channel acquisition (C=1): toggling the only channel off is allowed and yields black.
- toggle works on all three producers: after preview-only, after a MIP run, after reopen (D3).
- a stale/foreign cell is still ignored (`:583`, extends the existing coverage at `:193`).
- the recomposited RGB backing array outlives its QImage (OV11) — assert the canvas is still
  readable after a forced GC.
- memory bound at plate scale, mirroring `tests/test_montage.py:206`.

Export/HTML:
- the generated HTML lists one entry per channel and references the per-channel PNGs.
- the HTML shows the window each channel was exported at (D9 honesty requirement).

## Failure modes

| New codepath | Realistic production failure | Test? | Error handling? | User sees |
|---|---|---|---|---|
| store write on `add_tile` | tile shape ≠ cell shape after IMA-187 sub-grid | yes | corner-place by actual shape (`_montage.py:356-360`) | correct partial cell |
| toggle / re-window | full-res recomposite on GUI thread → freeze | yes (no-recompute) | display-res + coalescing (OV4) | responsive slider |
| contrast latch | running histogram stomps a manual window | yes | per-channel latch (D4) | window holds |
| reopen global window | early wells windowed by an immature histogram | yes | end-of-stream recomposite (OV7) | consistent plate |
| per-channel PNG export | a channel with no signal → flat/black PNG | yes | `_window` guards a flat channel (`_montage.py:144`) | black PNG, no crash |
| QImage backing array | numpy buffer freed under a live QImage | yes | keep a reference (OV11, `:1857`) | no segfault |
| C mismatch on reopen | well has fewer channels than omero lists | existing | raises loudly (`_montage.py:350`) | clear error |

No critical gaps: every failure mode above has both a test and error handling.

## NOT in scope

- **16-bit export + in-browser windowing** — D9 keeps HTML honest with toggles only; true
  re-windowing off a shared artifact is a real feature but a much bigger one. TODO'd.
- **Grayscale rendering for a single visible channel** — OV9 keeps LUT color to match the
  acceptance oracle; the grayscale preference is real but is a taste call for the users. TODO'd.
- **Multi-FOV sub-grid rendering itself** — that is IMA-187. This ticket consumes its cell shape.
- **Per-well (rather than global) contrast as a user option** — D6 settles on global because
  wells must stay comparable. A per-well mode is a different product decision.
- **Channel reordering / z-order in the composite** — additive compositing is order-independent.
- **Persisting channel/contrast state across sessions** — no requirement, and no settings store
  exists in the viewer today.
