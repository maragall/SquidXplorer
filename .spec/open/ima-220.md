# IMA-220 — Carrier background

_slug: carrier-background_

## Why
Draw the physical carrier like Squid NavigationViewer, so the plate view reads as a real
piece of labware: plate outline, skirt, A1 corner, and well positions you can orient against.

## Scope
Draw the carrier/wellplate PNG behind the region cells (Squid `software/images/*.png`),
calibrated so the artwork's wells land exactly on the widget's cells.

## Files
- `squidmip/_plate.py` — **new**: frozen per-format carrier calibration table (claimed early
  from IMA-214; that ticket wraps it in the Plate ABC later).
- `squidmip/images/*.png` — **new**: eight vendored carrier images.
- `squidmip/_viewer.py` — `PlateOverview` (`__init__`, `_canvas_for`, `_fit`, `_fit_cd`,
  `paintEvent`), `_plate_grid`, `PlateWindow._set_final`, and the two `PlateOverview(...)`
  call sites at `:1561` and `:1696`.

## Acceptance / oracle
The carrier PNG is drawn behind the regions and aligns with cell positions: for every
calibrated format, well A1's pixel in the PNG lands on the center of cell (0,0) and the
last well's pixel lands on the center of cell (nr-1, nc-1), at every zoom level.

## Depends on
plate-abc (IMA-214) — **relaxed**. IMA-220 needs only four numbers per format, so it lands
the calibration record in `_plate.py` itself rather than blocking on the full ABC.
See Decision D3.

# Engineering plan (locked by /plan-eng-review, 2026-07-20)

## The load-bearing insight

Squid's `NavigationViewer` (`control/core/core.py:1507-1656`) computes
`origin_x_pixel = a1_x_pixel - a1_x_mm / mm_per_pixel` because it must map a **live stage
position** onto the image. SquidMIP's `PlateOverview` is post-acquisition and addressed by
`(row, col)` — it never asks "where is the stage?". So `a1_x_mm` / `a1_y_mm` cancel out
entirely and the transform collapses to a pure ratio:

```
scale  = (cd / well_spacing_mm) * mm_per_pixel      # display px per PNG px
dest_x = ax + cd/2 - a1_x_pixel * scale             # A1's PNG pixel -> center of cell (0,0)
dest_y = ay + cd/2 - a1_y_pixel * scale
```

`cd` is the widget's existing displayed-px-per-well. This is why the existing edge-to-edge
cell grid is already **physically correct**: one cell is exactly one well pitch. No layout
rework is needed — only a background layer and a calibrated placement.

Consequence: IMA-220 needs just `(png, a1_x_pixel, a1_y_pixel, well_spacing_mm)` plus the
shared `mm_per_pixel`, not the stage-coordinate machinery.

## Calibration data (verified, Layer 1 reuse)

Vendored from Squid `objective_and_sample_formats/sample_formats.csv`. `mm_per_pixel` is
`0.084665` for the plate images (`core.py:1654`).

and `:1516` rejects anything else *before* `PlateOverview` is constructed. Vendoring the

| 1536   | `1536 well plate_1509x1010.png` | 130 | 93  | 2.25 | 32 | 48 |

Independently verified by the outside voice, which edge-detected every well center in all

**Not calibrated:** `glass slide` (CSV row is all zeros — `well_spacing_mm = 0`), the

## Implementation notes

**Alpha for the final montage without touching the worker.** `_set_final` (`:1857-1860`)
receives a 3-channel array covering *all* cells. Build the alpha there: allocate `(h, w, 4)`,
copy RGB into `[..., :3]`, set `[..., 3] = 0`, then set the `_CELL` block to 255 for each
rendered cell. Three defects the outside voice caught in the first draft of this note, all
now mandatory:

1. **Do not derive alpha from `_tiles_by_layer`.** It is never cleared except by rebuilding
   `PlateOverview`, so a `preview_limit` run followed by a full run accumulates into the same
   `_active_op_key`. It is also inconsistent by construction: `add_tile` drops tiles not in
   `self._by_rc` (`:583-584`, the stale-run guard) while `_final_montage` (`:884-895`) rebuilds
   from `self._raw` and does not. A mismatch marks a cell alpha 255 with RGB 0 — **opaque black
   squares sitting on the carrier**. Instead, `_set_final` takes an explicit `cells` argument:
   the exact set the montage was built from, passed by whoever built it.
2. **Bind the RGBA array to `self._final_arr`.** The `QImage` does not own its buffer. The
   array kept alive must be the *same* one the `QImage` points at — allocating a new `(h,w,4)`
   while `self._final_arr` still holds the old RGB array leaves the `QImage` reading freed
   memory. Assign `self._final_arr = rgba` before constructing the `QImage`.
3. **One format everywhere: `Format_ARGB32_Premultiplied`.** The first draft said
   `ARGB32_Premultiplied` for canvases and `RGBA8888` for the final — two formats flowing
   through one `_scaled` cache for no reason. Convert once, at the boundary.

Memory: the 1536wp final montage goes 35.7 MB -> 47.6 MB and briefly coexists with its RGB
source. Acceptable, but "no other consequences" was false — update the layer-stack comment at
`:536-538`, whose per-layer memory claim is now wrong.

**`add_tile` needs no change.** Drawing an opaque `RGB888` source onto an `ARGB32`
destination under the default `SourceOver` composition sets destination alpha to 255 in
that block automatically.

**Sub-pixel placement.** `carrier_placement()` returns floats, but `paintEvent` currently
places the montage with `int(ax), int(ay)` (`:701`). Drawing the carrier at an int-cast
destination lets it drift up to 1px from the grid and **jitter during pan**, while every unit
test on the pure function still passes. Use `QPointF`/`QRectF` for the carrier draw.

**Plumbing.** `PlateOverview.__init__` gains `carrier: CarrierSpec | None = None`. Only the
ingest call site at `:1561` passes one — it has `fmt` in scope from `:1515`. The computed-plate
site at `:1696` passes `None` per D10. Default `None` keeps every existing construction path
byte-identical.

**Inline ASCII diagrams** go in two places: the placement transform above, as a comment over
`carrier_placement()`; and the paint-order stack, as a comment at the top of `paintEvent`.
Update the layer-stack comment at `:536-538` — the per-layer memory claim changes with ARGB.

The placement math is extracted as a **pure function** `carrier_placement(cd, spec) ->
(scale, dest_x, dest_y)` precisely so the load-bearing logic is unit-testable without a
paint device. That is the primary seam.

Two tests exist because the obvious oracle is blind:

- **Orientation.** A well lattice is centro-symmetric, so "A1 lands on cell (0,0) and the last
  well lands on cell (nr-1,nc-1)" is satisfied *by construction* whether or not the artwork is
  upside-down. The outside voice measured 180°-rotation self-difference at 0.71/255 for 12wp —
  invisible to the eye as well. Squid renders through
  `addViewBox(invertX=not INVERTED_OBJECTIVE, invertY=True)` (`core.py:1555`), so the reference
  implementation *does* mirror on some configs and SquidMIP will not. Assert a known asymmetric
  pixel (the A1 corner chamfer).
- **Sub-pixel.** The pure function can be perfectly correct while `int()` truncation at the draw
  site makes the carrier jitter against the grid during pan. Needs a rendered-output assertion,
  not a unit test on the seam.

**Five regression tests are mandatory** (not optional per the review's regression rule),
because D2, D4 and D7 modify code that works today: `carrier=None` paint output, `add_tile`
opacity, final-montage display, `_fit`/`_fit_cd` geometry, and `_plate_grid` shape parity
with the deleted `_PLATE_DIMS`.

Tests go in `tests/test_viewer.py` (offscreen Qt is already configured at `:16`,
`QT_QPA_PLATFORM=offscreen`, with a `qapp` fixture at `:67`) and a new `tests/test_plate.py`
for the pure table/lookup paths, which need no Qt at all.

## Failure modes

| Codepath | Realistic production failure | Test? | Handled? | User sees |
|---|---|---|---|---|
| `carrier_for("glass slide")` | `ZeroDivisionError` from `well_spacing_mm = 0` | yes (D6) | yes — capability check | today's plate view, no crash |
| PNG missing from wheel | asset not packaged; `importlib.resources` raises | yes (D5) | yes — resolve test in CI | caught at CI, not at Nick's desk |
| carrier pixmap at max zoom | full-plate sizing allocates ~2.3 GB, OOM/freeze | yes (D11) | yes — viewport-cropped | would have been a hard hang |
| `_set_final` alpha vs cell set | preview-then-full run marks cells opaque with RGB 0 | yes | yes — explicit `cells` arg | opaque black squares on carrier |
| `_final_arr` rebinding | `QImage` reads freed memory -> garbage or segfault | yes | yes — assign before construct | crash or corrupt montage |
| reopen computed plate | no `wellplate_format` in `_meta` -> `KeyError` if assumed | yes (D10) | yes — passes `None` | no carrier, no crash |
| carrier orientation | artwork 180°-flipped; lattice symmetry hides it | yes | n/a — asserted, not handled | silently wrong plate picture |
| sub-pixel drift | `int()` truncation -> carrier jitters against grid on pan | yes | yes — `QPointF` | visible shimmer while panning |
| format `4` / `"4 glass slide"` | `:477` regex resolves both to `4`, no calibration | yes | yes — returns `None` | grid only, no carrier |

No critical gaps: every failure mode has both a test and an explicit handler. The orientation
case is the one that would be *silent* if untested, which is exactly why the asymmetric-pixel
assertion is mandatory rather than nice-to-have.

## Outside voice (Claude subagent, 2026-07-20)

Codex was unavailable (`not_installed`), so an independent subagent challenged the plan. It
**falsified nothing in the geometry** — it edge-detected every well center in all six PNGs and
confirmed the anchors, the `mm_per_pixel` constant, the one-cell-equals-one-pitch claim, and
that error does not grow toward the far corner. It found four things the review missed, all
folded in above: the unbounded carrier pixmap (D11), the missing format on the computed path
(D10), the `_SUPPORTED_PLATES` scope cut, and three concrete defects in the `_set_final` alpha
design. It also corrected the D4 fit math and caught the orientation and sub-pixel test holes.

**One recommendation rejected.** It argued D2's ARGB conversion is avoidable: color-key the
known `_BG` sentinel with `createMaskFromColor` at the single `_scaled` rebuild point, killing
a task and three regression tests. Rejected — any real tile pixel equal to `#070a0f` gets
punched transparent, and dark fluorescence backgrounds will produce exactly that. Correct-by-
construction beats correct-by-heuristic here, and color-keying is the kind of clever the house
style avoids. Its *implementation* corrections to D2 are accepted in full.

Factual slips it caught, corrected above: `6 well plate_1509x1010.png` is actually 1509x**1009**
(harmless — scale derives from anchors, not the filename); the `:477` regex maps `"4 glass
slide"` to `4`; and the `_PLATE_DIMS` regression baseline must be hardcoded, since T6 deletes
the dict it would otherwise compare against.

## Parallelization

Sequential implementation, no parallelization opportunity — every step lands in
`squidmip/` and the viewer changes are interdependent (the carrier cannot be seen until the
montage is transparent, and cannot be placed until the table exists).

Order: vendor assets + `_plate.py` table -> transparent montage (D2) -> placement + paint
(D4, D9) -> grid softening (D8) -> `_PLATE_DIMS` fold (D7) -> tests.
