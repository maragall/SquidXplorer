## What survives as IMA-211

### S1 — The acceptance oracle (the ticket's real, unclaimed value)

The original ticket said *prototype/evaluate* four stitchers. No other ticket owns evaluation,
and **IMA-222 has no acceptance criterion for stitch quality**. Evaluation does not require
shipping. Run the externals out-of-process on a dev box purely to produce reference output:

1. **Primary, runs today, needs no new acquisition:** cut one real 2084² plane into a grid at the
   true overlap with injected ±20 px offsets; require recovered offsets within 1 px and the
   composite **bit-exact** to the source outside feather zones.
2. **Seam metric on real data:** mean absolute difference across the seam ≤ 1.5× the same
   statistic measured inside a tile.
3. **Fallback accounting:** manifest reports per-well `registered` vs `nominal-fallback`; above a
   threshold, a loud warning — never a silent pass.
4. **External cross-check:** ASHLAR out-of-process for 3-5 wells; max displacement disagreement
   ≤ 2 px. This is the only independent signal available without ground truth.
5. **Negative test:** blank / uniform-noise overlap must fall back to nominal, never shift.

Why ASHLAR and not the others — verified 2026-07-20:

| Tool | Verdict |
|---|---|
| **MCmicro** | **Not a stitcher.** A Nextflow orchestrator that shells out to ASHLAR. Evaluating it measures ASHLAR twice. **Drop permanently.** |
| **ASHLAR** | The only importable Python option and the only one with a native plate/well concept (`--plates`, `PlateReader`, `plate=`/`well=` on `FileSeriesReader`). Derives positions from filename pattern + grid + overlap. **But `ashlar/reg.py` boots a JVM via pyjnius at import time**, so it is a dev-box tool, never a shipped dependency. |
| **BigStitcher** | JVM; no Python API (macro or Spark subprocess only, PyImageJ known-flaky); mandatory BigDataViewer XML+HDF5 resave; no plate/well concept. Second-most expensive. |
| **PetaKit5D** | MATLAB Compiler Runtime; `PyPetaKit5D` is Linux-only; SLURM defaults; hand-built `ImageList_*.csv` with an `axisOrder` string; 3D light-sheet throughout. Worst fit. |

### S2 — Geometry correction that contradicts IMA-187's lock ⚠

IMA-187 locked: *"coordinates.csv has NO `fov` column ... Mapping is row-order-per-region with a
loud cross-check against `fovs_per_region`."*

**That is working from an incomplete picture.** Verified by hand on
`~/Downloads/20x_scan_2025-09-05_17-57-50` — Squid writes **two** coordinate files:

```
coordinates.csv                                  region,x (mm),y (mm),z (mm)
                                                 └─ the PLANNED grid. No fov column. z blank.

original_coordinates/original_coordinates_{t}.csv  region,fov,z_level,x (mm),y (mm),z (um),time
                                                 └─ the ACTUAL stage position, WITH an explicit
                                                    fov key, per timepoint.
```

Consequences for IMA-187:
- The row-order mapping — the "silent wrong tile" hazard `docs/ima-189-eng-review.md` already
  refused once — becomes **unnecessary**. Cell assignment turns into arithmetic on an explicit key.
- IMA-187's own unresolved worry ("whether coordinates.csv carries one row per z-level on multi-z
  acquisitions, which would break the row-count cross-check") **dissolves**: `original_coordinates`
  carries `z_level` explicitly, so it can be filtered rather than guessed at.

**Second correction:** never derive grid geometry from `acquisition parameters.json`. On that same
dataset it reports `Nx=1, Ny=1, dx(mm)=0.9` while region `C5` holds **36 FOVs in a 6×6 grid at
0.7056 mm**. Any validator cross-checking against it refuses every real dataset. Derive spacing
from the coordinates themselves (modal positive delta of the sorted unique x per region).

### S3 — Scale facts (independently confirm IMA-187's headline number)

Measured on the 20x scan:

```
frame                2084 × 2084 uint16          =   8.7 MB
region C5            36 FOVs, 6×6, 0.7056 mm     ≈   9% overlap
composite side       5 × 1892 + 2084             ≈ 11,544 px
one composite plane  11544² × 2 B                ≈ 266 MB
per well (T=1, C=4)                              ≈ 1.07 GB
```

This independently reproduces IMA-187's locked "11541×11541 px = 1.07 GB at 4ch uint16" from a
different dataset and a different direction, which is good corroboration for its
**no-pixel-composition** decision.

It also means: **if IMA-222 ever does compose pixels**, `project_plate`'s
*"peak RSS ≈ workers × one well's footprint, flat in plate size"* (`_engine.py:187-193`) degrades
~40× while `workers` still defaults to CPU count and `_output` adds writer threads with a float32
pyramid transient. A workers cap for any fov-consuming operator is not an optimization.

### S4 — Two blockers that are not what the ticket says

The ticket says BLOCKED on Nick's laser-AF fix. More precisely:

- **The overlap fraction must be verified before anyone writes registration code.** Measured ~9%
  here; if Nick's acquisition runs 0% overlap, correlation is **dead on arrival** and only nominal
  placement works. This is cheap to check and gates real design decisions.
- **Focus quality is what the laser-AF block actually protects.** Out-of-focus tiles wreck phase
  correlation, so a stitch-quality test run against a broken-AF acquisition measures autofocus,
  not stitching. Running it early produces a misleading number, not an early answer.

Neither blocks S1's synthetic harness (oracle #1), which runs today.
