## Why

Evaluate **Odon as a fast local desktop viewer** for a region's projected output —
an alternative to the embedded `ndviewer_light` detail viewer — at arm's length
(GPL-3 clean: separate process, mere aggregation, binary never bundled).

**This ticket does NOT evaluate web or remote rendering.** Odon v0.1.5 is a native
Rust desktop GUI (`eframe`/`egui_glow`/`glow`, OpenGL): every code path except
`--check` calls `eframe::run_native` and opens an OS window. It has no HTTP server,
no WASM build, and no headless render-to-file. Its only listening socket is a
localhost MCP control bridge on `127.0.0.1:17870` that its own docs say must not be
exposed to a network. "Remote" in Odon means it can *read* zarr over HTTP/S3, not
that it *serves* pixels. Remote viewing remains an open question needing its own
ticket. (Decision 1A.)

## Phase 0 — Manual spike (GATE). ~30 min, human, zero code.

The ticket's purpose is to *evaluate*. The evaluation does not require the module;
the module requires the evaluation. Odon is not installed on this machine, ships
Apple-Silicon-only on macOS, and is 0.1.x with ~15 downloads. Building a permanent
module, a CLI flag, and ~20 tests into a BSD-3 package before anyone has looked at
Odon rendering our data inverts the risk. The plan's own reason for deferring the
GUI button — "commits user-visible surface to a 0.1.x dependency" — applies just as
hard to the CLI flag and harder to the tests.

Steps:

1. Install Odon from https://github.com/alexcoulton/odon/releases (macOS `.dmg`;
   Apple Silicon only).
2. Take an existing `<acq>.hcs/` from a previous `squidmip` run. Hand-write a ~20-row
   CSV next to `plate.ome.zarr`:
   ```csv
   id,path,well,fov
   B2_0,plate.ome.zarr/B/2/0,B2,0
   B3_0,plate.ome.zarr/B/3/0,B3,0
   ```
3. `odon --mosaic-samplesheet <that csv>` and look at it.

**Decide, and write the answer into this file before writing any code:**

| Question | Why it decides the ticket |
|---|---|
| Does the mosaic preserve plate geometry? | Odon has **no plate model** — the mosaic is a linear sequence in its own sort order. B2 will not sit left of B3 with row C beneath. For a plate viewer replacement this is plausibly **disqualifying**, and it costs 10 minutes to find out. |
| Is it actually faster / nicer than `ndviewer_light`? | The entire premise. If not, stop here. |
| How many ROIs before it degrades? | A 1536-well plate is 1536 separate zarr groups, each with its own GL textures and file handles. Nothing in Odon caps this. Find the practical ceiling; it determines whether `--odon` needs a row limit. |
| Do the pyramid levels align when zooming? | `_multiscales` computes `sy = p * (y0/y)`; odd-height fields are cropped before halving, so inter-level ratios are 2.0009…, not exactly 2.0. Odon uses `scale` for LOD placement, so misalignment would show as *drift while zooming*, not as an error. Length-checking the scale array (as the conformance test does) cannot catch this. Look for it. |
| Are the wrong channel colors tolerable? | See "Known limitation" below. |

**Gate:** if the mosaic is unusable for plate work, close IMA-212 with the finding
recorded and do not build Phase 1. That is a successful evaluation, not a failure.

## Phase 1 — the bridge (build ONLY if Phase 0 passes)

### What already exists — do NOT rebuild

`write_plate(reader, out_dir, regions=[...])` (`squidmip/_output.py:370`) already
writes per-region OME-Zarr, and every field directory
`plate.ome.zarr/{row}/{col}/{fov}/` is **already a valid Odon image group**. Verified
axis by axis against Odon's reader (`src/data/ome.rs`, `src/data/zarr_attrs.rs`):

| Odon requires | SquidMIP writes | |
|---|---|---|
| zarr v2 or v3 (`.zattrs`, else `zarr.json`) | zarr v3 | `_zarr_store.py:90` ✓ |
| NGFF 0.4 or 0.5; unwraps `{"ome": {…}}` | 0.5, nested under `ome` | `_zarr_store.py:92` ✓ |
| `multiscales[0]` present (hard error if absent) | always written | `_output.py:249` ✓ |
| every level: `scale` of exactly ndim length | 5 elements, 5 axes | `_output.py:180` ✓ |
| axes named `y` and `x` (hard error if absent) | t,c,z,y,x | `_output.py:184` ✓ |
| dtype uint8 or uint16 only (rejects float/int) | reader enforces exactly that set | `reader.py:55` ✓ |
| `shape.len() == axes.len()` | 5 == 5 | `_output.py:242` ✓ |

**So the pixel-writing half of this ticket is already built.** The new work is a CSV
writer plus a launcher. Writing a second copy of the pixels was rejected: it would
double an output already measured in hundreds of GB and create two sources of truth.
(Decision 2A.)

Odon has **zero HCS/plate support** — grepping its source for `plate`, `well`, `hcs`,
and `acquisition` returns no hits. Its samplesheet CSV is its own flattening
mechanism, so the plate becomes a row-per-field list with well identity demoted to a
metadata column.

### New module: `squidmip/_odon.py`

```
squidmip/_odon.py
├── write_samplesheet(hcs_dir) -> Path        # takes a DIRECTORY, not metadata
├── find_odon() -> Path                       # $ODON_BIN → PATH → platform default
├── check_odon(field_dir) -> bool             # headless conformance probe
└── launch_odon(samplesheet, *, mosaic_cols=None) -> Popen
```

#### REVERSAL — enumerate by directory glob, not by `select_fovs`

The first draft of this plan said "reuse `select_fovs`, do not re-walk the plate
directory," on DRY grounds. **That was wrong on two counts**, both caught by the
outside voice:

1. *It does not actually prevent disagreement.* Re-calling `select_fovs(metadata,
   n_fovs=…)` with independently hand-passed arguments is not the same as reusing
   the writer's result. `write_plate(reader, out, n_fovs=3)` followed by a default
   `write_samplesheet()` would silently emit a sheet covering a third of what was
   written, and the manifest returns `{plate, tiff, n_wells, n_fields_written,
   levels}` — no well list, no `n_fovs` — so there is nothing to cross-check
   against. The same trap applies to `--limit`, which `_cli.py:111-114` turns into
   `regions` for `write_plate` and which would have to be threaded through a second
   time.
2. *It forecloses the most likely real use.* A metadata-driven signature demands a
   live reader, so "I ran squidmip yesterday, open that output in Odon" is
   impossible. A directory walk makes it the default.

So:

```python
for zj in sorted(plate_dir.glob("*/*/*/zarr.json"), key=_sort_key):
    row, col, fov = zj.parent.parts[-3:]
```

Exactly three levels below `plate.ome.zarr` is precisely the field groups — the
plate, row, and well `zarr.json` files sit at depths 0, 1, and 2 and are not matched.
This cannot disagree with disk by construction, needs no `metadata`, no `n_fovs`, no
`regions`, and no separate exists-filter.

**Glob on `zarr.json`, never on the directory.** `_write_field` (`_output.py:241-252`)
writes the pyramid arrays first and calls `write_group` **last**, so a field
directory exists on disk before its `zarr.json` does. A run killed by `stop()` (the
GUI cancel path), a crash, or a full disk leaves a directory that exists with no
`multiscales` — which is exactly Odon's documented hard error. Testing
`field_dir.exists()` would pass that straight through.

`parse_well_id` (`_output.py:73`) is still reused, for **sort order only**: natural
column ordering (B2, B3, B10 — not B10, B2, B3) and the `(len(row), row)` row key.

#### CSV shape

```
id,path,well,fov
B2_0,plate.ome.zarr/B/2/0,B2,0
```

Header required; first two columns positionally `id,path`; paths **relative to the
CSV**, since Odon resolves relative paths against the samplesheet's own location and
absolute paths would break the moment the output is copied to a share. `row` and
`col` are dropped from the first draft — they are `parse_well_id(well)` and adding
them is redundant. Zero usable rows raises rather than emitting an empty CSV (Odon's
import fails on that anyway, with a worse message).

#### Entry point

A `--odon` flag on the existing `squidmip` CLI (`_cli.py`), a declarative pydantic
model where a new option costs ~4 lines. Because `write_samplesheet` takes a
directory, the flag also works standalone against a previously written `.hcs`.
**No GUI button yet** — deferred until Odon is actually adopted. (Decision 5A.)

## Data flow

```
squidmip <acq> --odon
   │
   ├─► write_plate(...)  [EXISTING, unchanged]
   │      └─► <acq>.hcs/plate.ome.zarr/{row}/{col}/{fov}/{0,1,..}
   │
   ├─► write_samplesheet(<acq>.hcs)                 [NEW]
   │      plate.ome.zarr.glob("*/*/*/zarr.json")    ← disk IS the source of truth
   │             │   (matches field groups only; plate/row/well sit shallower)
   │             │   (a partial write has no zarr.json yet → correctly excluded)
   │             ├─ parse_well_id(row+col) ──► natural sort key   [REUSED]
   │             └─► CSV row: id, relative path, well, fov
   │      └─► <acq>.hcs/odon_samplesheet.csv
   │
   └─► launch_odon()                                [NEW]
          find_odon() ──not found──► SystemExit w/ release URL + platform note
          │
          └─► Popen([odon, "--mosaic-samplesheet", csv])   detached
                 └─ poll() after ~1s: nonzero exit → warn loudly
                    (catches the bad-GPU/driver case; without this the process
                     dies silently and the user just sees no window)
```

## Binary discovery (Decision 4A)

Lookup order, first hit wins:

1. `$ODON_BIN` (explicit override; validated — a bad value errors rather than
   silently falling through, since an override that is quietly ignored is worse
   than none)
2. `shutil.which("odon")`
3. Platform default:
   - macOS: `/Applications/odon.app/Contents/MacOS/odon` — **the normal case.** The
     released `.dmg` installs an `.app` bundle and does *not* put `odon` on PATH, so
     PATH-only lookup would report "not installed" for most mac users.
   - Linux: `/usr/bin/odon` (from the `.deb`)
   - Windows: `%ProgramFiles%\odon\odon.exe`

Not found → one clear error naming https://github.com/alexcoulton/odon/releases and,
on `linux-aarch64`, stating plainly that **no Linux arm64 build exists**. Odon ships
exactly three artifacts: `odon-macos.dmg` (Apple Silicon only),
`OdonSetup-0.1.5-windows-x86_64.exe`, `odon_0.1.5_amd64.deb`. The spec's original
"Odon v0.1.5 is arm64" is true for macOS only.

**Version drift:** `--mosaic-samplesheet` and the positional `id,path` column
contract both come from 0.1.5 docs. `find_odon` proves a file exists, not that its
flags match — a 0.2.0 rename would surface as an opaque launch failure. Record the
verified-against version in the module docstring so the next failure is diagnosable.

**License boundary (load-bearing):** SquidMIP is BSD-3-Clause; Odon is GPL-3.0-only
(no "or later"). We launch a separately-installed, unmodified binary with CLI
arguments — mere aggregation, no copyleft reach. **Never vendor the binary, never
auto-download it, never add it to any installer.** Doing so would distribute GPL-3
code from a BSD package and void the arm's-length premise this ticket exists to test.

## Known limitation: Odon ignores our channel colors (Decision 6A)

Odon reads the `omero` block for channel **labels** and **contrast windows** but its
`OmeroChannel` struct has no color field at all. Colors come from a fixed eight-color
cycle in file order, plus one hardcoded case forcing any channel whose name contains
`dapi` to blue. Only channel 0 is visible by default.

So the same data renders in different colors in Odon than in `ndviewer_light` (which
*does* read `omero.color` and quantizes it to named colormaps). In multiplexed
fluorescence, color is the readout — this must be documented where an evaluator will
read it, or someone will either reject a good viewer for the wrong reason or, worse,
misread which marker is which. Document in the module docstring; **do not** rename
our channels to game Odon's `dapi` heuristic (encoding another tool's undocumented
hack into our output is the clever-over-explicit trap).

A regression test asserts we still write the resolved `display_color` into `omero`,
protecting `ndviewer_light`, which depends on it.

Odon also has **no `t` axis handling** — any axis that isn't c/z/y/x is pinned to
range `0..1`, so a T axis is silently fixed at timepoint 0. Harmless today (`n_t` is
1 in every current dataset) but Odon can never show a time series.

## Acceptance / oracle (Decision 3A — split, because Odon is only half-automatable)

The original one-line oracle is not runnable unattended: every samplesheet invocation
opens an OpenGL window and waits for a human. Odon's only headless path is
`odon --check <dataset.ome.zarr>`, which loads one coarse tile, prints
`OK: loaded tile level N …`, and returns before any window opens — but it is
single-dataset and **local-only**, and does not accept a samplesheet.

**Be honest about what the automated set proves.** It proves we emit a well-formed
CSV pointing at conformant groups. **Nothing automated proves Odon reads it** — that
is inherent to a GUI-only tool, and it is why Phase 0 is a gate rather than a
formality.

**Automated (gates the AFK build, Phase 1 only):**
1. `python -c "import squidmip"` succeeds; the existing suite passes (no regression).
2. CSV schema: header present; first two columns exactly `id,path`; no empty `id` or
   `path`; ≥1 usable row; every path relative and resolvable from the CSV's directory.
3. NGFF conformance **over the samplesheet's own rows** (not over the plate at large,
   which would pass with `_odon.py` absent): each row's path resolves to a directory
   whose `zarr.json` carries `attributes.ome.multiscales`, with a `scale` of exactly
   `len(axes)` per level, `y` and `x` among the axes, and dtype in `{uint8, uint16}`.
4. If the odon binary is present: `check_odon(first_field_dir)` returns True.
   `pytest.mark.skipif` when absent — it is not installed on this machine, and CI
   must not fail on an optional third-party tool.

**Manual (closes the ticket):** Phase 0's evaluation table, re-confirmed against the
generated samplesheet rather than the hand-written one.

| Failure | Test | Error handling | User sees |
|---|---|---|---|
| Binary not installed | ✓ | ✓ raise w/ URL | Clear message + install link |
| macOS .app not on PATH | ✓ | ✓ bundle fallback | Works silently |
| Linux arm64 | ✓ | ✓ explicit raise | "no Linux arm64 build exists" |
| Partial write: dir exists, no `zarr.json` | ✓ | ✓ excluded by glob | Row absent, Odon opens fine |
| Well skipped by `on_error` | ✓ | ✓ never globbed | Row absent |
| Output dir moved after CSV written | ✓ | ✓ relative paths | Works |
| Odon exits immediately (bad GPU/driver) | ✓ | ✓ `poll()` after ~1s | Loud warning, not silence |
| Odon renders wrong channel colors | ✓ | n/a — documented | Colors differ from ndviewer_light |
| Odon flag renamed in a future version | ✗ | partial | Opaque launch failure; version recorded in docstring to make it diagnosable |
| 1536 ROIs overwhelm Odon | Phase 0 | deferred | Measured in Phase 0; add a row cap only if Phase 0 shows one is needed |

**Critical gaps: 0.** The first draft accepted a silent Odon crash as "fire-and-forget
by design"; that was too quick — `poll()` after a short sleep is three lines and
converts it into a visible failure.

## NOT in scope

- **Web / remote rendering** — Odon cannot do it (see Why). Needs its own ticket.
- **GUI "Open in Odon" button** — deferred until Odon is adopted; `_viewer.py` is
  already 2081 lines. (Decision 5A.)
- **Second/flat OME-Zarr copy for Odon** — rejected, doubles disk. (Decision 2A.)
- **Symlink farm** — the samplesheet already flattens; also breaks on Windows.
- **Bundling or auto-downloading the odon binary** — GPL-3 boundary. (Decision 4A.)
- **Renaming channels to exploit Odon's `dapi` color hack** — clever over explicit. (6A.)
- **MCP/screenshot GUI automation** — needs a real GPU, cannot run in CI. (Decision 3A.)
- **A row cap / `--mosaic-cols` tuning** — deliberately unspecified until Phase 0
  measures where Odon actually degrades. Do not guess a limit.
- **Multi-timepoint** — Odon pins any t axis to 0; no current dataset has n_t > 1.
- **Fixing non-integer inter-level scale ratios** — a real property of `_multiscales`
  (odd axes cropped before halving), but out of scope here; Phase 0 checks whether it
  is even visible. If it is, that's a separate `_output.py` ticket.
- **Distribution changes** — this adds no new artifact; SquidMIP ships as before.
