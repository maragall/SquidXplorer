## BLOCKER — read this first

**IMA-222 exists to be "the slow registration+blend path, kept OUT of the fast mosaic
(IMA-187)." Only one of the two candidate stitchers actually does registration or blending,
and it is the one this repo has a standing decision not to depend on.**

Verified in code:

| | ian-stitcher (`image_stitcher`) | tilefusion |
|---|---|---|
| Registration | **None.** Only static `h_shift`/`v_shift` constants from scan-pattern params (`parameters.py:130-157`). | **Real.** `phase_cross_correlation` + SSIM scoring (`registration.py:21`, `:62`, `:123`) |
| Blending | **None.** `place_single_channel_tile` does a hard overwrite: `stitched_region[0, c, z, y0:y1, x0:x1] = tile_slice` (`stitcher.py:224`) | **Real.** Feather blend with weight profiles, `blend_numba_2d` (`fusion.py:269`) |
| Placement source | `coordinates.csv` | registered offsets |
| Packaged | No `pyproject.toml`; conda + `setup_ubuntu_22_04.sh` | Yes (hatchling, pip-installable) |
| Stitching CLI | Yes (`stitcher_cli.py`) | **No** — `[project.scripts]` is `stitcher-gui`, `convert-to-zarr`, `napari-zarr-viewer` |

So ian-stitcher is **naive coordinate placement**. Building IMA-222 on it would ship
IMA-187's fast mosaic wearing IMA-222's label — with a subprocess, a conda environment, and
a format handoff added on top. It would satisfy the literal acceptance text ("fuses a
region's FOVs and labels the plate stitched") while delivering none of the thing that makes
this ticket distinct from IMA-187.

And the conflict is not incidental: tilefusion's numba dependency — the reason this repo
banned it (`pyproject.toml:26-30`, and prior learning `tilefusion-init-heavy-import`) — **is
the blend kernel itself** (`fusion.py:11-17`, `@njit(parallel=True)`). "Avoid tilefusion" and
"ship a registration+blend operator" are in direct opposition. That tension was never
surfaced when the no-tilefusion rule was set, because at the time nothing needed blending.

### B1 — The decision that unblocks this ticket

**A) Do IMA-187 first, defer IMA-222.** Accept that naive placement is what is reachable
today. Ship the fast mosaic properly (in-process, vendored placement math is roughly 40
lines and needs no numba), and keep IMA-222 for when real registration is justified.
*Completeness 9/10. Honest, unblocks immediate multi-FOV value, no new deps.*

**B) Re-open the no-tilefusion rule for this operator only.** Depend on tilefusion behind a
lazy import inside the subprocess/worker so the heavy `__init__` never runs at app start.
Delivers the actual feature. *Completeness 8/10. Costs: a real numba/GPU/basicpy dependency
in the install story, and tilefusion has no stitching CLI, so a console-script must be added
upstream.*

**C) Build IMA-222 on ian-stitcher anyway.** Fastest to a green checkbox. *Completeness
3/10. Ships a mislabeled IMA-187 and burns the ticket. Not recommended.*

**Recommendation: A**, with the registry work (below) landing regardless. It is the only
option that does not either misrepresent what shipped or quietly reverse a standing
architectural decision without the wider context that set it.

## Why

Wells holding more than one FOV are currently sampled, not fused: `_viewer.py:1587-1589`
takes the first FOV per well and says so ("sampling 1 FOV/well (stitching TBD)"). A 2x2-FOV
well therefore shows a quarter of its area.

## What survives B1 regardless

These findings and tasks hold no matter which option is chosen.

### The Z=1 contract blocks stitch from being a registry peer

`projection.py:162` writes `out[t, c, 0] = reduce(planes)` — Z is pinned to size 1 — and
`_output.write_plate`, `_montage`, and `_viewer.py:817` (`image[0, :, 0]`) all depend on it.
The only operator seam is `reduce: Iterable[planes] -> plane`, a **z-reduction**. Stitch
consumes the FOV axis, not z, so registering it as a `_PROJECTORS` peer produces a shape the
writer cannot serialize, or silently drops the z-stack. Stitch therefore requires IMA-210's
consumes-axis registry to exist first. (IMA-225 deliberately shipped a pre-210 mechanism and
did not build 210 — so 210 is still unbuilt, and IMA-222 is its first genuine consumer.)

### The CLI promise

`_cli.py:10` states "a new operator is a new `--projector` value, not new CLI plumbing",
enforced by `_known_projector` at `:76`. An operator arriving as anything other than a
registry entry breaks that promise and needs its own CLI flag, or headless runs silently
cannot use it. Stitch needs an explicit decision here, not silence.

### Ordering: MIP-first is Nz× cheaper, but only commutes without blending

Stitch does not reduce z; a stitched result is `(1, C, Nz, H, W)`. Stitching all Nz planes
and then projecting throws away Nz−1 of them, so **MIP-first-then-stitch is 1/Nz the stitch
work and 1/Nz the intermediate disk**. Whether that is free depends on blending:

- **Hard overwrite (ian-stitcher):** placement is pure pixel selection, so it commutes with
  `max`. MIP-first is bit-identical and strictly cheaper. Take it.
- **Feather blending (tilefusion):** in overlap regions stitch-first gives
  `max_z(w_i·a_z + w_j·b_z)` while MIP-first gives `w_i·max_z(a_z) + w_j·max_z(b_z)`. These
  are **not equal** — max does not commute with a weighted sum. MIP-first changes pixels at
  seams, so it is a scientific choice, not an optimization.

This mirrors the IMA-225 precedent ("correction commutes with MIP — do it after at 1/Nz the
cost"), with the added caveat that the commutation fails once blending is on.

### Regressions to fix (mandatory, independent of B1)

1. **`_check_disk` under-estimates any fov-consuming operator.** `_viewer.py:1809` computes
   from `meta["frame_shape"]`; a fused region is about `n_fovs ×` that area and the
   pre-projection intermediate still carries `Nz`. Left alone the guard under-counts by
   roughly `n_fovs × n_z` and lets a run fill the disk it exists to protect.
2. **`_check_disk`'s message hardcodes "MIP"** (`_viewer.py:1817`) — already wrong for the
   `reference` projector today. Must derive from `Operation.label`.
3. **Stale ASCII diagrams.** `_engine.py:15-38` and `projection.py:10-19` both document a
   z-reduce-only pipeline and become actively misleading the moment a fov-consuming operator
   exists. Diagram maintenance ships with the change.

### Worker model (applies to any out-of-process option)

`_OperatorWorker` (`_viewer.py:773-839`) is built for an in-process generator yielding one
well at a time — that is where per-well progress, per-well fault isolation (`wellFailed`,
`_engine.py:236-239`), and Stop live. A blocking subprocess has none of them, so routing
stitch through it would leave those controls present in the UI but inert. Any out-of-process
option needs its own two-phase worker with:

- **kill-on-Stop and kill-on-close.** The existing terminal widget already learned this
  (`_viewer.py:386-393` kills and waits). Without it a stitch run outlives the app and keeps
  writing. **This is the one failure that is both silent and destructive — ship blocker.**
- **Partial-output cleanup.** Killing mid-write leaves a structurally-valid-looking but
  incomplete region store that a later read would happily consume as garbage. Needs an
  explicit incomplete-marker or cleanup-on-abort.

### Cross-tool disagreement risk (if any external stitcher is used)

- ian-stitcher **requires** `coordinates.csv` and merely `logging.warning`s then skips the
  timepoint when it is absent (`parameters.py:319-323`) — so it can produce **zero output
  with exit code 0** on an acquisition squidmip opens fine. squidmip deliberately never reads
  that file (`reader.py:29`, learning `squid-coordinates-no-fov-column`).
- Channel naming diverges: ian-stitcher does `replace("_"," ")` then `sorted(set(...))`
  (`parameters.py:343`, `:381`); squidmip resolves order from `acquisition_channels.yaml`.
  `_OperatorWorker._colors` is indexed **positionally** (`_viewer.py:804`), so a divergent
  order mis-maps display colors silently.
- squidmip's reader **rejects zarr outright** (`reader.py:106`,
  `NotImplementedError("Zarr layout detected")`) and its OME-TIFF path requires
  `{region}_{fov}` with a numeric fov (`_OME_STEM_RE`, `reader.py:263`), which
  `A1_stitched` fails. So re-reading any stitched output needs a **new reader adapter** —
  it is not a thin wrapper.
- Progress: `stitcher_cli.py` runs with `ProgressCallbacks.no_op()`, so the per-tile callback
  is dead in CLI mode. The only stdout signal is `logging.info("Processing region …")` —
  human prose, one tick per region. Real progress needs a `--progress-json` protocol upstream.
- Memory: squidmip's engine is explicitly bounded (`_engine.py:211-240`, "peak RSS ≈ N ×
  one-well footprint, FLAT in plate size"). ian-stitcher instead allocates the whole stitched
  region in RAM when it fits in 45% of available memory (`stitcher.py:101-117`). The plan's
  guard covers disk, not RAM, and cannot see inside a subprocess.

## Acceptance / oracle

Rewritten so it cannot be satisfied by the wrong implementation:

1. **The distinguishing test:** on a fixture whose true FOV offsets differ from the recorded
   stage coordinates by a known number of pixels, the operator's output aligns tile content
   to within a stated tolerance. Naive placement fails this; registration passes. *This is
   what separates IMA-222 from IMA-187, and the original oracle did not test it.*
2. MIP and `reference` are unaffected — still `consumes={"z"}`, outputs byte-identical.
3. On a single-FOV plate, output is byte-identical to a plain MIP run.
4. Stop ends the run with no orphan process and no reusable partial output.
5. `_check_disk` refuses a run that would not fit, with a message that does not say "MIP".
6. The plate title reads "… · Stitch" and the result is a re-openable OME-Zarr plate.
